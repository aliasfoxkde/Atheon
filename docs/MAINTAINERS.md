## How to reach a maintainer

1. **Public question / design discussion.** Open a GitHub
   Discussion under the matching category. The maintainers
   subscribe to all categories.
2. **Bug report / pattern submission.** Open a GitHub Issue
   using the template under `.github/ISSUE_TEMPLATE/`.
3. **Security disclosure.** Follow the process in
   [`.github/SECURITY.md`](../.github/SECURITY.md). Do **not**
   open a public issue.
4. **Maintainer-internal matters** (governance, conduct,
   personnel). Use the email listed for the specific
   maintainer. The BDFL's email is at the top of this file.

## What to expect

- **Issue triage:** within 7 days, usually faster.
- **PR first review:** within 14 days. If a PR is large or
  touches an unfamiliar subsystem, expect the first response
  to be a request for clarification rather than a substance
  review.
- **Security fixes:** fast-tracked. See
  [`.github/SECURITY.md`](../.github/SECURITY.md) for the SLA.

## Becoming a maintainer

See [`GOVERNANCE.md`](GOVERNANCE.md#becoming-a-maintainer).
In short: ship several high-quality PRs, get nominated by an
existing maintainer, and accept the role.
